## 1. 개요 및 주요 기능

**Figma MCP(Server)**는 AI 에이전트가 Figma Design / FigJam의 디자인 파일을 더 잘 이해하고, 경우에 따라 디자인 산출물을 직접 생성할 수 있도록 연결해주는 **MCP(Model Context Protocol) 기반 브리지**입니다.

기존 핵심은 그대로 세 가지로 볼 수 있습니다.

- **정방향 엔지니어링**: Figma 디자인 → 코드 생성
    - 예: Figma 프레임을 읽고 React, Next.js, Tailwind CSS 컴포넌트로 구현
    - `get_design_context`, `get_screenshot` 같은 도구가 핵심 입력으로 사용됨
- **역방향 엔지니어링**: 기존 웹 서비스(URL) / 실행 중인 웹앱 → Figma 디자인 레이어로 변환
    - 예: 로컬 Next.js 앱 또는 운영 중인 웹사이트를 캡처해 Figma의 편집 가능한 레이어로 생성
    - `generate_figma_design` 중심의 워크플로우
- **다이어그램 생성**: Mermaid 구문 → FigJam 다이어그램 자동 생성
    - 예: 시스템 아키텍처, 플로우차트, 시퀀스 다이어그램 등을 FigJam에 생성
    - `generate_diagram` 사용

즉, Figma MCP의 본질은 단순히 “Figma 파일을 읽는 도구”가 아니라, **디자인과 코드 사이의 왕복 루프를 AI가 수행할 수 있게 하는 연결 계층**입니다.

<aside> 💡

핵심 관점: Figma MCP는 디자이너와 개발자 사이의 수동 핸드오프를 줄이고, AI가 **디자인 컨텍스트 → 코드 구현 → 다시 디자인 반영**까지 이어지는 루프를 만들 수 있게 합니다.

</aside>

## 2. Local vs Remote 구분

Figma MCP를 이해할 때 가장 먼저 구분해야 하는 것은 **Local MCP 서버**와 **Remote MCP 서버**입니다. 같은 “Figma MCP”라고 해도, 어떤 서버/클라이언트를 쓰는지에 따라 사용 가능한 도구가 달라질 수 있습니다.

|구분|설명|주요 용도|
|---|---|---|
|**Local / Desktop MCP**|로컬 환경 또는 데스크톱 기반으로 Figma 컨텍스트를 MCP 클라이언트에 전달|디자인 읽기, 코드 구현용 컨텍스트 추출|
|**Remote MCP**|Figma의 원격 MCP 서버를 통해 링크 기반으로 파일/노드 컨텍스트를 가져오거나 일부 생성 작업 수행|`whoami`, `generate_figma_design`, Claude Code 기반 역방향 워크플로우|

특히 아래 기능은 조건을 명확히 확인해야 합니다.

- `whoami`: **Remote only**
- `generate_figma_design`: **Remote Figma MCP server only + Claude Code only**
- 일부 쓰기/생성 기능은 클라이언트마다 노출 여부가 다를 수 있음

<aside> ⚠️

“Figma MCP가 연결되어 있다”와 “모든 Figma MCP 도구를 사용할 수 있다”는 다릅니다. Remote/Local 여부, Claude Code 사용 여부, Figma 플랜/Seat, 파일 권한에 따라 실제 도구 목록이 달라질 수 있습니다.

</aside>

## 3. 주요 도구(Tools) 및 기능 상세

기존에 정리했던 “주요 도구” 섹션을 공식 문서 기준에 맞춰 조금 더 정확하게 보강하면 아래와 같습니다.

|도구명|분류|기능 설명|비고|
|---|---|---|---|
|**`get_design_context`**|Read|특정 Figma 파일 또는 노드의 구조화된 디자인 컨텍스트를 가져옵니다. 레이아웃, 계층 구조, 스타일, 제약 조건 등을 코드 생성에 활용합니다.|디자인 → 코드 변환 시 핵심|
|**`get_screenshot`**|Read|특정 노드의 시각적 참조 이미지를 가져옵니다. 텍스트 기반 메타데이터만으로 판단하기 어려운 여백, 정렬, 시각적 의도를 보완합니다.|정확도 향상용|
|**`get_metadata` / 유사 메타데이터 도구**|Read|파일의 전체 구조, 노드 목록, 메타정보를 빠르게 파악하는 데 사용됩니다.|환경/클라이언트에 따라 명칭 차이 가능|
|**`generate_figma_design`**|Write / Generate|웹앱 UI 또는 URL 기반 화면을 Figma의 편집 가능한 레이어로 역변환합니다.|Claude Code only + Remote only|
|**`generate_diagram`**|Generate|Mermaid 구문을 기반으로 FigJam에 다이어그램을 생성합니다.|아키텍처/플로우 문서화에 유용|
|**`get_variable_defs` / 변수 관련 도구**|Read|디자인 토큰(색상, 간격, 타이포 등)의 변수 정의를 확인해 코드에 반영합니다.|디자인 시스템 연동에 중요|
|**`whoami`**|Identity|현재 인증된 Figma 사용자 정보를 확인합니다.|Remote only, 권한/쿼터 확인에 유용|
|**`get_code_connect_suggestions`**|Code Connect|Figma 컴포넌트와 코드베이스의 실제 컴포넌트 간 매핑 후보를 제안합니다.|디자인 ↔ 코드 컴포넌트 동기화|
|**`send_code_connect_mappings`**|Code Connect|추천된 Code Connect 매핑을 확정/전송하는 흐름에 사용됩니다.|컴포넌트 매핑 확정|

<aside> 📝

기존 메모의 `Youtube` 항목은 공식 Tool 이름이라기보다는 영상/가이드 콘텐츠를 도구명처럼 적어둔 것으로 보입니다. 역할상으로는 “파일 구조와 노드 목록을 빠르게 파악하는 메타데이터 조회 도구”에 가까우므로, 문서에서는 `get_metadata` 또는 “메타데이터 조회 계열”로 정리하는 편이 더 정확합니다.

</aside>

## 4. 링크 기반 사용 방식: 노드 선택 → URL → MCP

Figma MCP의 대표적인 사용 흐름은 **Figma에서 특정 프레임/레이어를 선택하고, 그 URL을 AI에게 전달하는 방식**입니다.

1. Figma Design에서 구현하고 싶은 프레임/레이어를 선택
2. 브라우저 주소창 또는 “Copy link to selection”으로 선택 노드 링크 복사
3. Claude Code 같은 MCP 클라이언트에 링크와 작업 지시 입력
4. 에이전트는 URL에서 **file key / node id**를 추출
5. MCP 서버가 해당 노드의 디자인 컨텍스트를 반환
6. 에이전트가 컨텍스트를 기반으로 코드 작성 또는 분석 수행

<aside> 🧩

URL은 사람이 보는 링크이면서 동시에 AI가 **파일 키와 노드 ID를 추출하기 위한 입력 포맷**입니다. 따라서 “파일 전체 링크”보다 “선택한 프레임/레이어 링크”가 더 정확한 결과를 만듭니다.

</aside>

## 5. 권장 워크플로우: Figma 디자인 → 코드 변환

AI에게 디자인 분석과 코드 생성을 요청할 때는 아래 순서가 효율적입니다.

1. **`get_design_context`**
    - 전체 디자인 컨텍스트 파악
    - 레이어 계층, 스타일, 오토레이아웃, 컴포넌트 구조 확인
2. **메타데이터/노드 목록 확인**
    - 응답이 너무 크거나 전체 파일이 복잡한 경우, 필요한 특정 노드만 필터링
    - 구현 대상 범위를 좁혀 불필요한 tool call 감소
3. **`get_screenshot`**
    - 시각적 참조 이미지 확보
    - 실제 보이는 결과와 구조화된 컨텍스트 간 차이를 보정
4. **`get_variable_defs` / 디자인 토큰 확인**
    - 색상, 간격, radius, typography 등 변수 기반 스타일 확인
    - Tailwind config, CSS variable, design token 매핑에 활용
5. **코드 생성**
    - React / Next.js / Tailwind CSS / shadcn/ui 등 프로젝트 스택에 맞춰 구현
    - 컴포넌트 분리, 상태 관리, 반응형 처리까지 포함
6. **검증 루프**
    - 구현 결과와 screenshot을 비교
    - 불일치 영역만 추가 컨텍스트 요청
    - tool call을 최소화하면서 정확도 개선

### 프롬프트 예시: 디자인 → React + Tailwind

```
아래 Figma 링크의 선택 프레임을 React + Tailwind CSS로 구현해줘.

요구사항:
1. Auto Layout 구조를 최대한 반영해줘.
2. Button, Input, Card, Modal 단위로 컴포넌트를 분리해줘.
3. 색상/타이포/간격은 디자인 토큰 또는 CSS variable로 매핑해줘.
4. 애매한 시각 정보는 get_screenshot으로 확인한 뒤 구현해줘.
5. 최종적으로 컴포넌트 파일 구조와 코드 예시를 함께 제안해줘.
```

## 6. 역방향 워크플로우: `generate_figma_design`

기존 웹사이트나 실행 중인 프론트엔드 프로젝트를 Figma 디자인으로 옮기고 싶을 때 사용하는 강력한 기능입니다.

- **작동 방식**
    1. URL 입력
    2. 브라우저 캡처/분석
    3. 레이어 구조 추론
    4. Figma의 편집 가능한 디자인 레이어 생성
- **출력 모드(outputMode)**
    - `newFile`: 새로운 Figma Design 파일 생성
    - `existingFile`: 기존 Figma 파일 내 특정 위치에 추가
    - `clipboard`: 클립보드에 복사 후 Figma에서 붙여넣기
- **중요 조건**
    - `generate_figma_design`는 **Claude Code only**
    - **Remote Figma MCP server only**
    - 기존 파일에 쓰려면 해당 파일에 대한 **편집 권한** 필요

<aside> ⚠️

이 기능은 “웹 화면을 이미지로 붙여넣는 것”이 아니라, 가능한 범위에서 Figma의 **편집 가능한 레이어 구조**로 변환하는 것을 목표로 합니다. 다만 HTML/CSS 구조가 복잡하거나 커스텀 컴포넌트가 많은 경우, 완벽한 의미 단위 컴포넌트로 복원되지는 않을 수 있습니다.

</aside>

## 7. 실습 예시: Next.js 프로젝트 역캡처

정방향/역방향 실습은 목적이 다릅니다.

- **정방향(Design → Code)**: 이미 존재하는 Figma 디자인을 읽고, React / Next.js / Tailwind 코드로 구현
- **역방향(Code/Web → Design)**: 이미 구현된 웹 UI를 캡처해, Figma의 편집 가능한 디자인 레이어로 재구성

공식 문서 기준으로 `get_design_context`는 기본적으로 **React + Tailwind 형태의 구현 힌트**를 제공할 수 있고, 프롬프트를 통해 프레임워크/스타일 시스템을 커스터마이즈할 수 있습니다. 반대로 `generate_figma_design`는 **라이브 웹앱/사이트 UI를 Figma Design 레이어로 보내는 기능**이며, 로컬 UI는 직접 캡처 흐름이 가능하고, 운영/스테이징 사이트는 Playwright 같은 브라우저 자동화 도구와 함께 쓰는 방식이 언급됩니다.

<aside> 🧪

실습은 한 번에 전체 서비스를 대상으로 하기보다, **Hero / Pricing Card / Modal / Dashboard Widget**처럼 작은 단위부터 테스트하는 것이 좋습니다. 그래야 tool call을 아끼고, 결과 품질도 더 쉽게 검증할 수 있습니다.

</aside>

### 7.1 정방향 예시: Figma Pricing Card → Next.js + Tailwind 컴포넌트

#### 목표

Figma에 있는 `Pricing Card` 프레임을 선택하고, 이를 실제 Next.js 프로젝트의 재사용 가능한 컴포넌트로 구현합니다.

#### 예시 상황

- Figma 선택 노드: `Pricing Card / Pro Plan`
- 구현 대상: `src/components/pricing/PricingCard.tsx`
- 기술 스택: Next.js, React, TypeScript, Tailwind CSS
- 기대 결과:
    - 카드 레이아웃 구현
    - 가격/플랜명/혜택 목록/CTA 버튼 분리
    - 색상/간격/타이포는 디자인 토큰 또는 Tailwind theme 기준으로 매핑
    - 모바일/데스크톱 반응형 처리

#### 단계 1. Figma에서 구현할 프레임 선택

Figma에서 `Pricing Card` 프레임을 선택한 뒤 **Copy link to selection**으로 선택 노드 링크를 복사합니다.

```
<https://www.figma.com/design/{fileKey}/{fileName}?node-id=123-456>
```

#### 단계 2. Claude Code / Cursor 등에 구현 요청

```
아래 Figma 선택 프레임을 Next.js + TypeScript + Tailwind CSS 컴포넌트로 구현해줘.

Figma URL:
{선택한 Figma 노드 URL}

요구사항:
1. get_design_context로 레이어 구조, 오토레이아웃, 텍스트 스타일, 색상 정보를 먼저 확인해줘.
2. 시각적으로 애매한 spacing, radius, shadow는 get_screenshot으로 확인해줘.
3. 결과 코드는 src/components/pricing/PricingCard.tsx에 들어갈 수 있는 형태로 작성해줘.
4. Button은 기존 src/components/ui/button.tsx를 재사용해줘.
5. 색상/spacing은 하드코딩하지 말고 Tailwind theme 또는 CSS variable을 우선 사용해줘.
6. props는 planName, price, description, features, ctaLabel, highlighted 정도로 분리해줘.
```

#### 단계 3. 예상 Tool 흐름

```
1. get_design_context
   - 선택 노드의 계층 구조 확인
   - Auto Layout 방향/간격/padding 확인
   - 색상, typography, radius, shadow 확인

2. get_screenshot
   - 실제 시각 결과 확인
   - context만으로 애매한 정렬/간격 검증

3. 변수/토큰 확인
   - 가능하면 Figma variable 또는 design token 매핑

4. 코드 작성
   - React 컴포넌트 생성
   - Tailwind class 또는 프로젝트 디자인 시스템으로 변환
```

#### 단계 4. 산출물 예시 구조

```
src/
  components/
    pricing/
      PricingCard.tsx
  app/
    pricing/
      page.tsx
```

#### 단계 5. 결과 검증 체크리스트

- 카드 width / padding / gap이 Figma와 유사한가?
- 버튼 높이, radius, hover 상태가 디자인 시스템과 맞는가?
- 텍스트 hierarchy(플랜명, 가격, 설명, feature)가 정확한가?
- highlighted plan의 border/background가 Figma와 일치하는가?
- 모바일에서 카드가 깨지지 않는가?

#### 추가 프롬프트: 구현 후 보정

```
방금 구현한 PricingCard.tsx를 Figma screenshot과 비교해서,
spacing / typography / border radius / shadow 기준으로 다른 부분을 찾아 수정해줘.
수정 이유를 항목별로 설명하고, 최종 코드를 다시 제시해줘.
```

### 7.2 정방향 예시: Figma Dashboard 화면 → 컴포넌트 분해

복잡한 화면은 한 번에 구현하기보다, Figma 화면을 **섹션/컴포넌트 단위로 분해**한 뒤 순차 구현하는 방식이 좋습니다.

#### 예시 대상

- `Dashboard Page`
    - Sidebar
    - Top Navigation
    - KPI Cards
    - Chart Area
    - Recent Activity Table

#### 프롬프트 예시

```
아래 Figma Dashboard 프레임을 바로 코드로 만들기 전에,
먼저 컴포넌트 분해 설계를 해줘.

Figma URL:
{Dashboard 프레임 URL}

요구사항:
1. get_design_context로 전체 레이아웃 구조를 파악해줘.
2. 구현 단위를 Sidebar, Header, KPI Card, Chart Panel, Activity Table로 나눠줘.
3. 각 컴포넌트의 props 설계를 제안해줘.
4. 실제 구현 순서를 추천해줘.
5. 이후 1단계로 Sidebar만 먼저 구현해줘.
```

#### 왜 이렇게 나누는가?

- 전체 화면을 한 번에 구현하면 tool call과 코드 수정 범위가 커집니다.
- 작은 단위로 나누면 screenshot 비교와 QA가 쉬워집니다.
- 기존 코드베이스의 `Button`, `Card`, `Table`, `Chart` 컴포넌트를 재사용하기 좋습니다.

### 7.3 역방향 예시: 로컬 Next.js 앱 → Figma Design 레이어

#### 목표

이미 구현된 Next.js 화면을 Figma로 가져와, 디자이너가 수정 가능한 레이어 구조로 확인할 수 있게 합니다.

#### 예시 대상

- 본인 사이트: `goddaehee.co.kr`
- 또는 로컬에서 실행 중인 Next.js 프로젝트
- 예시 URL: `http://localhost:3000`

#### 단계 1. 로컬 서버 실행

```bash
npm run dev
# 또는
pnpm dev
```

실행 후 브라우저에서 아래 URL이 정상 동작하는지 확인합니다.

```
<http://localhost:3000>
```

#### 단계 2. 캡처 범위 결정

처음부터 전체 사이트를 캡처하기보다, 범위를 좁히는 것이 좋습니다.

- `/` 홈 화면 Hero 영역
- `/pricing` 가격 카드 섹션
- `/dashboard` KPI 카드 영역
- `/login` 로그인 폼

#### 단계 3. Claude Code에서 `generate_figma_design` 요청

```
현재 로컬에서 실행 중인 Next.js 앱의 <http://localhost:3000/pricing> 화면을 Figma 디자인으로 변환해줘.

요구사항:
1. 새 Figma Design 파일로 생성해줘.
2. Header / Pricing Hero / Pricing Card Grid / FAQ 섹션이 가능한 한 별도 프레임 또는 그룹으로 나뉘게 해줘.
3. 텍스트는 이미지가 아니라 편집 가능한 텍스트 레이어로 유지해줘.
4. 버튼, 카드, 배지 같은 UI 요소는 재사용 가능한 구조로 정리해줘.
5. 단순 screenshot이 아니라 편집 가능한 Figma 레이어를 목표로 해줘.
```

#### 단계 4. 예상 Tool 흐름

```
1. 로컬 URL 접근
2. 화면 렌더링 상태 확인
3. generate_figma_design 실행
4. outputMode에 따라 새 파일 / 기존 파일 / 클립보드 중 하나로 결과 전달
5. Figma에서 레이어 구조 확인
```

#### 단계 5. Figma 결과물 확인 체크리스트

- 텍스트가 실제 텍스트 레이어로 분리되었는가?
- Header / Section / Card / Button 단위가 어느 정도 구분되었는가?
- 레이어 이름이 이해 가능한가?
- 이미지/아이콘이 적절히 분리되었는가?
- Auto Layout 또는 프레임 구조가 후속 편집에 적합한가?
- 반응형 상태가 필요하다면 desktop/mobile을 별도 캡처했는가?

<aside> ⚠️

`generate_figma_design` 결과는 “완벽한 디자인 시스템 복원”이 아니라, **라이브 UI를 편집 가능한 Figma 레이어로 가져오는 출발점**에 가깝습니다. 복잡한 CSS, animation, canvas, absolute positioning이 많은 화면은 수동 보정이 필요할 수 있습니다.

</aside>

### 7.4 역방향 예시: 운영 웹사이트 → Figma로 가져오기

운영/스테이징 URL을 대상으로 할 때는 로컬보다 변수가 많습니다. 로그인, 쿠키, 동적 로딩, 반응형 viewport, 네트워크 지연 등을 고려해야 합니다.

#### 예시 프롬프트

```
아래 운영 웹사이트의 랜딩 페이지를 Figma 디자인으로 캡처해줘.

URL:
<https://example.com>

요구사항:
1. desktop viewport 기준으로 먼저 캡처해줘.
2. Hero / Feature / Pricing / Footer 섹션을 구분해줘.
3. 가능하면 새 Figma 파일로 생성해줘.
4. 동적 로딩이 끝난 뒤 캡처해줘.
5. 결과물이 단순 이미지가 아니라 편집 가능한 레이어인지 확인해줘.
```

#### 운영 사이트 캡처 시 주의점

- 로그인 뒤 화면은 인증 세션이 필요할 수 있음
- lazy loading으로 인해 일부 이미지/섹션이 누락될 수 있음
- 반응형 확인을 위해 desktop / tablet / mobile viewport를 나눠 캡처하는 것이 좋음
- 외부 폰트/아이콘이 Figma에서 다르게 보일 수 있음

### 7.5 왕복 루프 예시: 웹 → Figma → 코드 개선

Figma MCP의 강점은 한 방향 변환이 아니라 **왕복 루프**입니다.

```
1. Next.js로 UI 구현
2. generate_figma_design으로 현재 UI를 Figma에 캡처
3. Figma에서 디자이너가 레이아웃/간격/카피 수정
4. 수정된 Figma 프레임을 get_design_context로 다시 읽음
5. 변경 사항을 코드에 반영
```

#### 왕복 루프 프롬프트 예시

```
방금 생성된 Figma 디자인과 현재 Next.js 코드 구현을 비교해줘.

비교 기준:
1. 레이아웃 구조
2. spacing / padding / gap
3. typography scale
4. color token
5. button/card component 재사용성
6. mobile responsive 대응

결과:
1. 차이점을 표로 정리해줘.
2. 수정 우선순위를 P0/P1/P2로 나눠줘.
3. 실제 코드 수정안을 제안해줘.
```

이렇게 하면 **Figma 디자인 → 코드 구현 → 라이브 UI 캡처 → Figma 보정 → 코드 재반영**의 양방향 루프를 만들 수 있습니다.

## 8. 플랜 및 쿼터 주의사항

Figma MCP는 플랜/Seat에 따라 tool call 한도가 다릅니다. 특히 Starter 플랜은 매우 제한적일 수 있습니다.

- Starter 플랜은 월 **6 tool calls** 수준으로 제한될 수 있음
- `whoami`, `get_design_context`, `get_screenshot`, `generate_figma_design` 등 각각의 호출이 tool call로 누적될 수 있음
- 한 번의 실험 과정에서도 여러 번 호출되면 쿼터가 빠르게 소진될 수 있음

### 쿼터를 아끼는 방법

- 파일 전체가 아니라 **선택 프레임/노드 링크**를 전달
- “전체 구현” 대신 “Modal 영역만”, “Header만”처럼 범위를 좁힘
- 처음부터 screenshot을 남발하지 않고, 애매한 경우에만 호출
- 반복 실험 전 `whoami` / 플랜 / 연결 계정을 먼저 확인

<aside> 📉

실습할 때는 “한 번에 전체 사이트 캡처”보다 **작은 화면 단위로 쪼개서 테스트**하는 편이 쿼터 관리와 결과 품질 모두에 유리합니다.

</aside>

## 9. 자주 겪는 이슈 / 트러블슈팅

### 9.1 `invalid node id` 오류

가능한 원인:

- 선택 노드 링크가 아니라 파일 전체 링크를 전달함
- 해당 노드가 삭제되었거나 이동됨
- 권한이 없는 파일/팀의 링크를 사용함
- Remote/Local 연결이 꼬였거나 인증이 만료됨

해결:

- Figma에서 다시 프레임을 선택하고 “Copy link to selection”
- 더 상위 프레임 단위로 링크 복사
- Figma MCP 재인증
- 파일 접근 권한 확인

### 9.2 `generate_figma_design`가 보이지 않음

가능한 원인:

- Claude Code가 아닌 클라이언트를 사용 중
- Remote MCP가 아니라 Local MCP만 연결됨
- 클라이언트가 해당 tool을 아직 노출하지 않음
- 플랜/권한 조건 불충족

해결:

- Remote Figma MCP 설정 확인
- Claude Code 환경에서 다시 확인
- MCP tool list 확인
- Figma 플랜/Seat/파일 위치 확인

### 9.3 결과물이 “의미 있는 컴포넌트”로 안 나옴

가능한 원인:

- 웹 UI가 semantic HTML이 아니라 div 중첩 위주로 구성됨
- CSS-in-JS / animation / canvas / 복잡한 absolute layout 사용
- 디자인 시스템과 코드 컴포넌트 매핑 정보 부족

개선:

- HTML 구조와 접근성 태그 개선
- 컴포넌트 네이밍 정리
- Figma 컴포넌트와 코드 컴포넌트를 Code Connect로 매핑
- 디자인 토큰과 CSS token을 1:1로 맞춤

## 10. 실전 프롬프트 모음

### 10.1 Figma → 코드

```
이 Figma 링크의 선택 프레임을 Next.js + Tailwind CSS 기준으로 구현해줘.
디자인 컨텍스트와 스크린샷을 모두 참고하고,
컴포넌트 분리 기준과 파일 구조까지 제안해줘.
```

### 10.2 특정 노드만 구현

```
전체 화면이 아니라 Modal 영역만 구현해줘.
필요한 경우 해당 노드의 screenshot을 가져와 시각적으로 검증한 뒤 코드 작성해줘.
```

### 10.3 디자인 토큰 반영

```
Figma 변수/디자인 토큰을 확인해서 색상, spacing, radius, typography를 Tailwind theme 또는 CSS variable로 매핑해줘.
하드코딩은 최소화해줘.
```

### 10.4 웹 → Figma

```
<http://localhost:3000> 화면을 Figma 디자인으로 변환해줘.
새 Figma 파일로 생성하고, Header / Hero / Card / Footer 단위로 레이어가 잘 나뉘도록 해줘.
```

### 10.5 생성된 Figma → 코드 개선

```
생성된 Figma 디자인과 현재 코드베이스를 비교해서,
디자인과 다른 부분을 찾아주고,
컴포넌트 구조/스타일 토큰/반응형 처리 개선안을 제안해줘.
```

## 11. 요약 및 결론

- **Figma MCP**는 디자이너와 개발자 사이의 간극을 줄여주는 혁신적인 도구입니다.
- 핵심 흐름은 다음 세 가지입니다.
    - **디자인 → 코드**
    - **웹/코드 → 디자인**
    - **텍스트/Mermaid → FigJam 다이어그램**
- 단순히 디자인을 보는 것을 넘어, **웹을 디자인으로, 디자인을 코드로** 자유롭게 변환하는 양방향 루프를 만들 수 있습니다.
- 다만 실제 사용 시에는 **Remote/Local 구분, Claude Code 전용 기능, 플랜/쿼터, 파일 권한**을 반드시 확인해야 합니다.
- Claude Code 같은 터미널 기반 AI와 결합하면 특히 강력하며, Next.js / React / Tailwind 프로젝트와 함께 사용할 때 생산성이 높습니다.
